SECTION 01

SaaS 시스템을 건물 구조로 이해하기

AIMS SaaS는 4개 층으로 나뉩니다. 우리가 개선하려는 것은 인테리어(프론트엔드)뿐입니다.

프론트엔드 — 사용자가 보는 화면
대시보드, 메뉴, 버튼 등 사용자가 직접 클릭하는 모든 것
🏠 건물 내부 인테리어

화면의 요청을 받아서 처리하고, 결과를 돌려주는 엔진
🏠 건물의 배관 · 전기 · 통신 시설
MongoDB / Azure Storage — 데이터
태그 정보, 매장 정보, 이미지 등 모든 데이터 보관

핵심 포인트

이번 개편은 맨 위층(프론트엔드)의 방 배치만 바꾸는 것입니다. 아래 3개 층(백엔드, DB, 인프라)은 전혀 건드리지 않습니다.

Multi-Tenant 구조

하나의 건물을 여러 고객사(리테일러)가 함께 사용합니다. A고객 데이터는 B고객에게 보이지 않는, 논리적으로 분리된 구조입니다.

IoT Hub란?

매장의 Gateway 장비와 클라우드 서버를 연결하는 전용 통신 채널입니다. 장비 상태 확인, 펌웨어 업데이트 등을 담당합니다.

Critical 서비스 간 데이터가 연결되지 않음

ESL, Gateway, Vision 등의 데이터들이 분산 되어있어, 데이터가 서로 연결되지 않습니다.

예를 들어 "A매장에서 ESL에서 배터리 문제가 발생했을 때, 그 원인이 사용 환경 때문인지, 배터리 불량인지, 교체 주기 도래인지" — 지금 구조에서는 알 수 없습니다.

💡 비유: 5개 부서가 있는 회사인데, 각 부서가 서로 다른 메신저를 쓰고, 회의록도 공유하지 않는 상태. 전사 회의를 해도 전체 그림이 안 그려지는 상황.

Major 목적이 다른 UX가 한 화면에 혼재

"디바이스 관리"(ESL·GW 상태 조회)와 "선반 모니터링"(Vision·Loadcell 결품 감지)이 같은 메뉴 레벨에 섞여 있습니다.

사용자가 "결품 상태를 보려고" 들어왔는데 ESL, LCD, LBS, Gateway를 다 지나쳐야 Vision 메뉴를 찾을 수 있습니다.

💡 비유: 원룸에 주방·침실·서재·창고를 칸막이 없이 다 때려넣은 상태. 요리하려면 침대 옆을 지나가야 하고, 책 찾으려면 냉장고를 열어야 하는 격.

Strategic 대시보드 정보 부족 — 의사결정 지원 불가

현재 Dashboard는 "Device Attention" 수준에 그쳐 있어, 실질적인 매장 운영 인사이트를 제공하지 못합니다.

  • ✕ 역할/권한별 화면 분리 기능 없음
  • ✕ Asset Management 항목 부재
  • ✕ Health Monitoring 항목 부재
  • ✕ 고객사별 환경 차이 미반영
💡 비유: 자동차 계기판에 속도계만 있고, 연료계·엔진온도·타이어압력이 없는 상태. 운전은 할 수 있지만, 정보에 기반한 판단이 불가능.

SECTION 04

기술적으로 가능한 이유

이미 검증된 3가지 근거를 바탕으로, 프론트엔드 재배치가 기술적으로 실현 가능합니다.

🔀

프론트엔드 분리 가능

AIMS SaaS는 Vue.js로 되어 있고, 새 Sub App은 다른 기술 스택으로 만들어도 됩니다. 사용자는 같은 브라우저 안에서 메뉴만 전환하면 됩니다. SSO로 로그인 한 번이면 모든 Sub App 접근 가능.

Vue.js + SSO (Single Sign-On)
🔗

데이터 연동 검증 완료

AIMS SaaS는 이미 Open API를 제공 중입니다. Sub App이 필요한 데이터(Shelf Group ID, Store ID 등)는 API로 단방향 참조만 하면 되어, 양방향 동기화가 불필요합니다.

REST API · 단방향 참조 · Webhook
🛡️

백엔드·인프라 변경 없음

Azure AKS(서버), IoT Hub(장비 통신), MongoDB(데이터), Image Generator(이미지 생성) — 기존 인프라는 모두 그대로 유지됩니다. 온톨로지용 Neo4j만 Sub App 전용으로 신규 추가.

Azure AKS · IoT Hub · MongoDB 유지
